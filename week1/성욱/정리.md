### 1.1 책에 대하여

자바스크립트 정복이 목적지가 되어서는 안되고 여행이 되어야한다.

시간을 들여 책의 내용을 완전히 내 것으로 만들자

### 1.2 자바스크립트 이름의 유래

마케팅 목적으로 자바와 비슷하게 지었다.

초반에는 '모카'라는 암호명으로 불렸다.

ECMA 표준 기구에 의한 명칭은 'ECMAScript'이다.

### 1.3 명세서

TC39는 Javascript를 관리하는 기술 운영 위원회로, Javascript의 공식 명세를 관리한다.

위원회는 약 50 ~ 100명으로 구성되고 웹에 투자를 많이 하는 회사 출신이 주를 이룬다.

모든 제안은 5단계로 이루어지고 https://tc39.es/process-document 에서 확인할 수 있다.

모든 제안은 https://github.com/tc39/proposals 에서 확인 가능하다.

모든 주요 브라우저, 디바이스 제조사는 단 하나뿐인 명세서를 기준으로 Javascript 구현체를 만든다.

### 1.3.1 JS를 지배하는 웹

대부분 명세서에 정의된 JS와 브라우저 엔진에서 돌아가는 JS는 동일하다.

불일치가 발생하면 TC39 위원회가 기존 결정을 철회하고 명세서를 웹에 맞춘다.

ex) contains() -> includes()

명세서를 변경하지 않는 경우도 있는데 웹에서만 쓰이는 경우에 주로 그렇다.

ex) escape(), unescape(), anchor(), blink(), compile()

### 1.3.2 JS지만 JS가 아닌 웹 전용 문법

alert()는 명세서에 없지만 모든 웹 환경에 지원하는 함수다

부록 B에는 이와 관련된 내용이 없는데 Node.js 같은 환경도 전역 스코프에 API를 추가해 자체적으로 기능을 제공하는 경우도 있기 때문이다.

### 1.4 JS의 다양한 얼굴

다양한 패러다임 - 절차적, 객체 지향적, 함수형 패러다임

절차적 - 코드가 톱다운이면서 선형적으로 구조화되고 이 때 프로시저라 불리는 코드 단위에 미리 정해진 일련의 연산을 작성한다

객체 지향적 - 클래스 기준으로 코드를 구조화하며, 클래스에는 로직과 데이터가 정의된다

함수형 - 부수 효과가 없는 순수 함수 단위로 코드를 구조화한다. 함수 자체가 값으로 취급된다는 특징이 있다

모든 패러다임이 JS에 적용 가능하다

### 1.5 하위 호환성과 상위 호환성

JS는 하위 호환성 보장이며 상위 호환성은 보장되지 않는다

HTML, CSS는 상위 호환성이 보장되며 하위 호환성이 보장되지 않는다

하위 호환성 - 단 한번이라도 유효한 문법이라고 인정되면 명세서가 변경되더라도 절대 그 유효성이 깨지지 않는다는 의미

상위 호환성 - 새로운 명세서에 추가된 문법으로 코드를 작성했을 떄 이전 명세서를 준수하는 구형 JS 엔진에서 문제가 발생하지 않는다는 의미

### 1.5.1 간극을 줄이기 위한 노력

구 엔진과 호환되지 않는 문법은 트랜스파일을 통해 문제를 해결 (주로 바벨을 사용)

굳이 트랜스파일러를 쓰는 이유 - 최신 문법으로 클린한 코드를 짜기 위함

### 1.5.2 간극을 메우기 위한 방법 찾기

폴리필 - 지원하지 않는 메서드를 해결하기 위해 메서드 정의를 추가해주는 해결책

ex) Promise.prototype.finally

### 1.6 인터프리터 이해하기

JS는 소스 코드를 배포하지, 바이너리 파일을 배포하는게 아니기 때문에 컴파일 언어가 아니라고 많이 주장한다

모든 컴파일 언어는 '파싱'을 거친다.

JS로 작성한 소스 코드는 실행 전에 파싱을 거친다

-> 파싱을 거치는데 그렇다면 컴파일 언어이기도 한 걸까? -> 답은 '예'에 가깝다.

JS 실행 단계

1. 바벨이 트랜스파일. 웹팩같은 번들러를 거쳐 번들링, JS 엔진에 전달
2. JS 엔진은 코드를 파싱해 추상 구문 트리로 바꾼다.
3. 엔진은 추상 구문 트리를 이진 바이트 코드로 바꾼다. JIT(Just-in-time) 컴파일러가 작동하며 최적화가 함께 진행된다
4. JS 가상 머신이 프로그램을 실행한다.

### 1.6.1 웹어셈블리

Wasm은 JS가 주력이 아닌 개발자도 JS 엔진에서 돌아가는 코드를 쉽게 작성할 수 있게 해주는데 목적이 있었다.

실행 전 단계인 파싱과 컴파일을 거치지 않아 ASM.js와는 다르다.

Wasm은 어셈블리 언어와 유사하며 배포는 JS 엔진의 별도 처리가 많이 필요하지 않은 바이너리 파일 형식으로 진행된다.

Wasm을 통해 고 언어로 작성한 프로그램을 JS 엔진이 해석할 수 있도록 한다면 JS 언어 자체가 스레드를 지원하지 않아도 괜찮지 않냐는 사례가 있다.

Wasm은 웹과 직접적인 연관이 없다. JS 엔진에서 실행되긴 하지만 정적 타입 정보에 크게 의존하기 때문에 JS는 Wasm 프로그램을 짜는데 가장 적합하지 않은 언어다.

Wasm을 이용하여 다른 언어가 JS를 대체하길 원하는 사람들이 있다.

### 1.7 엄격 모드

엄격 모드를 사용하면 손실보다 이득이 크다

JS 엔진이 코드를 최적화하고 효율적으로 실행할 수 있게 해주는 최고의 '안내 가이드 역할을 하는 모드'이다.

엄격 모드에서만 활성화되는 가이드 대부분은 **초기 오류**의 형태를 띠는데, 초기 오류는 코드 실행 전 컴파일 단계에서 잡아낼 수 있는 오류를 의미한다.

### 1.8 정리

JS는 ECMA 주최하에 TC39 위원회가 결정하는 ECMASCript 표준을 구현한 언어다

JS는 다중 패러다임 언어다

JS는 컴파일 처리되는 언어다
