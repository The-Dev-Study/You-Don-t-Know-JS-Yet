# 인터프리터 이해하기

JS는 인터프리터 언어일까 컴파일 언어일까?  
JS는 소스코드 자체를 배포하지, 바이너리 파일을 배포하는 게 아니기 떄문에 많은 사람들이 JS는 컴파일 언어가 아니라고 주장함
(인터프리터 언어인지 컴파일러 언어인지에 따라 배포 방식이 달라짐)

스크립트 언어나 인터프리터 언어는 대개 위에서 아래로 한 줄씩 코드가 실행  
그리고 보통 실행이 시작 전 거치는 사전 단계가 없음
예) 5번째 줄에 오류가 있다면 4번째 줄이 실행되기 전까지 오류 발견할 수 없음

실행이 되기 전 파싱이라 부르는 사전 단계를 거치는 언어  
5번째 줄 오류가 있다면 파싱 단계에서 오류가 발견, 사전에 오류 차단 가능  
-> 굳이 실행하지 않아도 파싱이 되지 않아 프로그램 실행할 수 없기 떄문  
-> 실행 전 오류가 포함된 코드 조각이 어디있는지 알 수 있기 때문에 구문오류(syntax error | static error) 차단에 유리

파싱을 거치는 언어와 컴파일을 거치는 언어의 공통점

- 모든 컴파일 언어는 파싱을 거침
  - 파싱 -> 파싱 결과인 추상구문 트리(AST: abstract syntax tree)를 컴퓨터가 실행할 수 있는 형태로 변경
  - 파싱을 거치는 언어는 파싱에서 끝나는 게 아니라 실행 가능한 코드를 생성하는 작업까지 수행  
    -> 따라서 파싱을 거치는 언어는 컴파일 언어라 통용되는 것

JS로 작성한 코드는 실행 전 파싱을 거침

1. 코드 작성 완 -> 바벨이 트랜스파일 -> 웹팩 등 번들러를 거쳐 번들링 -> 그 결과가 JS엔진에 전달
2. JS엔진이 전달받은 코드를 파싱해 추상 구문 트리로 바꿈
3. JS엔진이 추상 구문 트리를 이진 바이트 코드로 바꿈(이 과정에서 JIT 컴파일러 작동, 최적화가 함께 진행)
4. JS 가상 머신이 프로그램 실행

그렇다면 JS는 컴파일 언어인가??  
예... 개발자가 이상한 문법을 입력하는 등 실수를 하더라도 코드 실행 전 정적 오류를 미리 발경할 수 있는 특징이 있기 때문
